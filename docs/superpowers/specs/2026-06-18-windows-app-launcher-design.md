# Windows App Launcher — дизайн

## Цель

Создать production-ready приложение для Windows на Tauri v2, React и Tailwind CSS. Оно сканирует установленные программы и ярлыки меню «Пуск», показывает их с нативными иконками, фильтрует по имени и запускает через Windows Shell.

## Технологии

- Tauri v2 и Rust для нативной части.
- React, TypeScript и Vite для интерфейса.
- Tailwind CSS для оформления.
- Zustand для состояния.
- `winreg`, `walkdir`, `windows`, `image`, `base64`, `serde` и `serde_json` для Windows-интеграции, изображений и кэша.
- Vitest и Testing Library для frontend-тестов; встроенный test harness Rust для backend-тестов.

## Архитектура

Приложение состоит из React-интерфейса и модульного Rust-бэкенда внутри Tauri v2. Rust-модули разделяют сканирование источников, извлечение иконок, работу с кэшем и запуск приложений. Тяжёлые операции выполняются через `tauri::async_runtime::spawn_blocking`.

Backend предоставляет команды `get_apps`, `refresh_apps` и `launch_app`. `get_apps` сначала читает JSON-кэш из `app_data_dir`. Если кэш существует, команда немедленно возвращает его и запускает фоновое сканирование. После завершения backend атомарно обновляет кэш и отправляет событие `apps://updated`. Без кэша команда дожидается первого сканирования. `refresh_apps` всегда выполняет полное сканирование, обновляет кэш и возвращает список.

## Поиск приложений

Сканер читает три ветки реестра Windows:

- `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall`;
- `HKLM\SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\Uninstall`;
- `HKCU\Software\Microsoft\Windows\CurrentVersion\Uninstall`.

Также рекурсивно сканируются общая и пользовательская папки программ меню «Пуск». Для записей реестра используются `DisplayName`, `DisplayIcon` и `InstallLocation`. Запись принимается только при наличии имени и разрешимого пути к существующему `.exe` или `.lnk`. Строки `DisplayIcon` очищаются от кавычек, аргументов и индекса ресурса иконки. Из `InstallLocation` выбирается подходящий исполняемый файл без деинсталляторов и вспомогательных бинарников.

Обновления Windows, системные компоненты, деинсталляторы и записи без запускаемого пути исключаются. Дубликаты объединяются по канонизированному пути без учёта регистра. Итог сортируется по имени без учёта регистра. Стабильный `id` формируется из нормализованного пути.

## Иконки и запуск

Для `.exe` и `.lnk` иконка извлекается через Win32 Shell API. `SHGetFileInfoW` получает `HICON`; пиксели считываются через GDI в BGRA, преобразуются в RGBA, кодируются в PNG и возвращаются как `data:image/png;base64,...`. Все Win32-ресурсы освобождаются даже при ошибке. Если иконку извлечь нельзя, `icon_base64` равен `null`.

Запуск `.exe` и `.lnk` выполняется только через `ShellExecuteW` с операцией `open`. Значение результата проверяется по правилам Win32; ошибки преобразуются в понятное сообщение для интерфейса.

## Интерфейс

Выбрано направление «Воздушный Slate»: тёмный градиентный фон, синий акцент, полупрозрачные поверхности, тонкие светлые границы и умеренно крупные карточки.

Header содержит название, количество приложений, поиск и кнопку обновления. Поиск фильтрует карточки в реальном времени без повторного обращения к backend. Основная область использует адаптивную сетку. Карточка показывает иконку, обрезанное название и полный путь во всплывающей подсказке. При отсутствии иконки используется нейтральная иконка `AppWindow` из Lucide.

Первичная загрузка отображает skeleton-карточки. Пустой результат поиска показывает отдельное состояние. Запуск и успешное обновление подтверждаются toast-уведомлением; ошибки отображаются как error toast. Карточки имеют короткие fade-in и hover-анимации с учётом `prefers-reduced-motion`.

## Состояние и поток данных

Zustand store хранит полный список, поисковый запрос, статус первичной загрузки, статус обновления и последнюю ошибку. Компоненты получают уже отфильтрованный список через селектор. При событии `apps://updated` store атомарно заменяет список свежими данными. Подписка Tauri снимается при размонтировании корневого компонента.

Сбой отдельной записи реестра, ярлыка или иконки не прерывает общее сканирование. Фатальные ошибки команды возвращаются через `Result` и показываются пользователю. Повреждённый кэш игнорируется и заменяется результатом полного сканирования.

## Тестирование

Rust unit-тесты покрывают очистку путей, фильтрацию мусорных записей, дедупликацию, сортировку и чтение/запись кэша. Код, зависящий от Win32 и реестра, отделяется от чистых преобразований, чтобы критические правила тестировались без изменения системы.

Frontend-тесты покрывают первичную загрузку, поиск, пустое состояние, обновление списка, обработку backend-события и ошибку запуска. Перед завершением выполняются `npm test`, `npm run build`, `cargo test` и `cargo check`.

Ручная проверка на Windows подтверждает сканирование реальных приложений и `.lnk`, отображение иконок, запуск через Windows Shell, обновление кэша и горячую перезагрузку интерфейса.

## Локальный запуск и ярлык

В корне проекта создаётся PowerShell-скрипт запуска. Он проверяет наличие Node.js, npm и Rust, устанавливает npm-зависимости при отсутствии `node_modules` и запускает `npm run tauri dev`. Ярлык на рабочем столе открывает этот скрипт в видимом окне PowerShell. Vite hot reload показывает изменения React/CSS без перезапуска; изменения Rust автоматически пересобирают Tauri-приложение.

## Границы первой версии

Первая версия предназначена только для Windows. Она не включает категории, избранное, ручное добавление программ, удаление приложений, автообновление бинарника и телеметрию.
